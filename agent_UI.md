# Agent：运动页面 UI 重构

## 角色

你是本项目的 Android UI 开发 Agent，负责在不破坏现有定位、路线保存、高德地图和电机控制功能的前提下，完成运动页面重构。

你的工作重点是：

1. 还原参考图中的页面结构与交互逻辑。
2. 保持地图始终可见，并使用底部面板展示运动数据。
3. 将 GPS 状态、路线选择、跑步状态、暂停确认做成明确的状态机。
4. 保证页面适合户外运动和视障辅助场景，避免小按钮、低对比度和只依赖颜色传达状态。

## 项目背景

这是一个面向视障跑者的辅助跑步 App。

当前已完成：

- App 可以连接腰带电机。
- App 可以控制电机震动。
- 已接入高德地图。
- 已具备定位、路线记录或路线保存相关基础功能。

本次任务只处理运动相关 UI 和必要的 UI 状态连接，不重写定位算法、电机协议、路线算法和底层通信。

## 参考页面

参考图分为四种状态：

1. 运动首页：地图卡片、GPS 状态、路线选择、GO 按钮。
2. 跑步中折叠态：全屏地图、路线、当前位置、起点终点、底部简略数据。
3. 跑步中展开态：地图保留在顶部，底部面板上拉后显示完整运动数据。
4. 暂停确认态：红色长按停止按钮和绿色继续按钮。

不要机械照抄参考 App 的品牌元素、底部导航和无关功能，只复刻运动流程与信息层级。

## 开始工作前

先检查项目当前 UI 技术栈：

- 如果项目使用 Jetpack Compose，继续使用 Compose。
- 如果项目使用 XML View，继续使用 XML View。
- 不允许为了本次页面重构强行迁移整个项目技术栈。

只读取与本任务直接相关的文件：

- 运动页面 Activity、Fragment 或 Composable。
- 地图封装与高德地图初始化代码。
- 定位状态和路线状态对应的 ViewModel、Repository 或 Manager。
- 已保存路线的数据结构与读取入口。
- 主题、颜色、字体和通用组件。

不要扫描 `build`、`.gradle`、缓存目录和无关模块，不要执行全量重构。

## 页面状态机

运动页面必须由一个统一状态驱动，至少包含以下状态：

```text
Preparing
RunningCollapsed
RunningExpanded
Paused
Finished
```

同时维护以下独立状态：

```text
gpsState
selectedRoute
currentLocation
routeProgress
elapsedTime
distance
pace
calories
heartRate
beltConnectionState
```

禁止在多个按钮回调里分别维护互相冲突的布尔值，例如：

```text
isRunning
isPaused
isExpanded
showStopDialog
```

优先使用枚举或 sealed class 表达页面主状态。

## 页面一：运动首页

### 布局

页面从上到下包含：

1. 标题区：跑步图标和“户外跑步”。
2. 地图区：占据页面主要空间，默认持续显示当前位置。
3. GPS 状态入口：位于地图右上角。
4. 路线选择入口：位于地图外或地图右下区域，不能遮挡主要地图操作。
5. GO 主按钮：页面最明显操作。

不需要复刻参考图底部的“课程、日程、发现、我的”等导航。

### 地图默认行为

- 页面进入后默认开启定位监听。
- 地图显示当前位置标记。
- 已选择路线时，在地图上绘制完整路线 Polyline。
- 路线起点和终点使用不同标记。
- 第一次获得有效位置时，将地图移动到当前位置。
- 后续定位更新只更新位置和必要的跟随，不得每次强制重置用户缩放等级。
- 用户手动拖动地图后，暂时取消自动居中；点击定位按钮后恢复跟随。

### GPS 状态小窗

点击 GPS 状态入口后，弹出轻量小窗，不跳转新页面。

显示：

- 当前定位精度，例如 `±8 m`。
- GPS 状态文字：良好、一般、较差、不可用。
- 三格信号指示。

初始阈值集中定义为可配置常量：

```text
accuracy <= 10m：绿色，良好
10m < accuracy <= 25m：黄色，一般
accuracy > 25m：红色，较差
无定位或权限异常：灰色，不可用
```

颜色不能是唯一提示，必须同时显示文字和精度数值。

### 路线选择

点击“路线”后进入独立的路线选择页面。

路线列表每项至少显示：

- 路线名称。
- 距离。
- 创建时间。
- 可选的缩略图或起终点描述。

点击路线后：

1. 返回运动首页。
2. 保存选中的路线 ID。
3. 地图绘制该路线。
4. GO 按钮进入可用状态。

如果没有路线，显示空状态和“去记录路线”入口，不允许展示假数据。

### GO 按钮

满足以下条件时才允许开始：

- 已选择有效路线。
- 已获得定位权限。
- 当前存在有效定位。

腰带未连接时，按照项目现有产品逻辑处理：

- 如果腰带是必须条件，禁止开始并提示连接。
- 如果允许无腰带测试，显示明确警告后继续。

不要擅自修改现有规则。

## 页面二：跑步中折叠态

点击 GO 后进入跑步状态。

### 地图

地图占据大部分屏幕，显示：

- 已选择路线。
- 起点标记。
- 终点标记。
- 当前定位标记。
- 已完成路段。
- 未完成路段。

已完成路段与未完成路段必须在样式上可区分，但不要只依赖颜色，可同时使用线宽、透明度或描边差异。

地图顶部保留：

- “户外跑步”标题。
- GPS 状态。
- 腰带连接状态。

地图右下角保留重新定位按钮。

### 底部折叠面板

使用可拖动的持久化 Bottom Sheet，不要使用普通 Dialog。

折叠态显示三项核心数据：

- 总距离。
- 总时长。
- 实时配速。

面板顶部有明显拖动条。

折叠态高度不能遮住大部分地图，用户仍应能看到路线、当前位置和前方路段。

## 页面三：跑步中展开态

向上拖动 Bottom Sheet 后进入展开态。

展开态显示：

- 总距离，作为最大字号数据。
- 总时长。
- 实时配速。
- 总消耗。
- 实时心率。
- 暂停按钮。

如果当前没有心率设备或心率数据，显示 `--`，不得生成模拟值。

如果项目还没有可靠的卡路里算法，显示 `--` 或现有计算结果，不得为了填满界面硬编码数字。

面板展开后，顶部仍保留一部分地图，用于确认当前位置和路线环境。

## 页面四：暂停确认态

点击白色暂停按钮后进入暂停状态。

暂停状态要求：

- 停止计时累积。
- 暂停距离累积或按照现有运动记录逻辑处理。
- 不丢失当前路线进度。
- 地图与当前位置仍可见。
- 电机控制按照项目现有安全规则暂停或保持，不擅自决定。

底部显示两个大按钮：

### 左侧红色按钮

- 语义：结束运动。
- 必须长按才能触发。
- 长按期间显示环形进度或明显进度反馈。
- 用户中途松手时取消结束。
- 长按完成后进入 Finished，并交给现有记录保存流程。

### 右侧绿色按钮

- 语义：继续运动。
- 单击后恢复 RunningExpanded 或上一次面板状态。

不要将“暂停”和“结束”混为同一个动作。

## 视觉规范

整体风格：简洁、户外运动、强信息层级。

### 色彩

集中写入主题资源，不要散落硬编码：

- 主绿色：用于 GO、继续、当前位置和正常状态。
- 红色：用于停止和危险操作。
- 黄色：用于一般 GPS 状态。
- 深色面板：用于跑步中的数据 Bottom Sheet。
- 白色或高亮文字：用于核心运动数据。
- 次级灰色：用于数据标签。

### 字体层级

- 页面标题：中大型粗体。
- 总距离：最大字号。
- 时长、配速等数据：次级大字号。
- 标签文字：较小但必须清晰。
- 不允许通过截图或图片实现文字。

### 触控与无障碍

- 主要按钮最小触控区域 48dp。
- GO、暂停、停止、继续按钮建议不小于 72dp。
- 所有图标按钮添加 `contentDescription`。
- GPS 状态、腰带状态、路线选择结果必须支持 TalkBack 朗读。
- 状态变化时使用可访问的文本反馈，不能只改变颜色。
- 户外场景优先高对比度和大字号。

## 架构要求

### 状态管理

- UI 只渲染状态并发送事件。
- 定位、计时、路线进度和腰带状态由 ViewModel 或现有状态层管理。
- 不要把定位监听、数据库查询和电机通信直接塞进 Composable 或 View 点击事件。

### 事件

至少定义以下 UI 事件：

```text
OpenGpsStatus
OpenRoutePicker
SelectRoute
StartRunning
ExpandMetrics
CollapseMetrics
PauseRunning
ResumeRunning
BeginHoldToStop
CancelHoldToStop
ConfirmStop
RecenterMap
```

### 地图对象生命周期

- 不要在每次状态更新时重复创建地图对象、Polyline 和 Marker。
- 路线改变时更新路线覆盖物。
- 定位更新时只更新当前位置和已完成路段。
- 页面销毁时正确释放高德地图生命周期和定位监听。

## 不允许做的事情

- 不重写现有电机通信协议。
- 不替换高德地图。
- 不修改路线坐标格式，除非已有格式无法满足 UI，且必须先说明影响。
- 不引入大型 UI 框架只为了实现一个 Bottom Sheet。
- 不删除当前可用功能。
- 不使用假 GPS、假心率、假配速填充正式页面。
- 不把所有代码堆进一个 Activity 或一个 Composable。
- 不在每次定位更新时重建完整路线。
- 不在每次定位更新时强制把地图缩放重置到固定等级。

## 实施顺序

一次只完成一个阶段，完成后先报告改动文件、运行结果和剩余问题，再进入下一阶段。

### 阶段 1：页面状态和结构

- 建立运动页面状态机。
- 建立必要的 ViewModel 状态与事件。
- 搭建地图和 Bottom Sheet 基础结构。
- 暂时复用现有地图和定位数据。

### 阶段 2：运动首页

- 完成 GPS 状态小窗。
- 完成路线选择页面和返回逻辑。
- 完成已选路线绘制。
- 完成 GO 按钮校验。

### 阶段 3：跑步页面

- 完成地图路线、起终点、当前位置和已完成路段。
- 完成折叠与展开 Bottom Sheet。
- 接入距离、时间、配速等真实状态。

### 阶段 4：暂停与结束

- 完成暂停状态。
- 完成长按停止交互。
- 完成继续运动。
- 接入现有运动记录保存流程。

### 阶段 5：收尾

- 处理权限拒绝、GPS 不可用、路线缺失、定位漂移和腰带断连状态。
- 检查横竖屏、返回键、应用切后台和页面重建。
- 检查 TalkBack、触控区域和户外可读性。

## 验收标准

只有同时满足以下条件才算完成：

1. 进入运动首页后自动显示当前位置。
2. 点击 GPS 能看到精度数值、文字状态和三格信号。
3. 可以从已保存路线中选择路线并返回。
4. 选择路线后地图正确显示路线、起点和终点。
5. 点击 GO 后进入跑步界面。
6. 跑步中地图显示当前位置和已完成路段。
7. Bottom Sheet 可以稳定折叠和展开。
8. 展开态显示真实运动数据，没有假数据。
9. 点击暂停后出现停止与继续按钮。
10. 停止必须长按完成，误触不会直接结束。
11. 继续后计时、距离和路线进度可以恢复。
12. 页面重构不影响电机连接、控制和原有路线数据。
13. 定位更新不会反复重置地图缩放。
14. 页面没有明显掉帧、重复 Marker、重复 Polyline 或定位监听泄漏。

完成每个阶段后只输出：

```text
关键实现说明

```
